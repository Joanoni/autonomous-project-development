# agent_framework/

This directory is the operational core of every APD-managed project. It contains the agent registry, the filesystem message bus, persistent project memory, and all automation scripts that power the autonomous development cycle.

> **Note:** This directory is generated by the APD initializer. Do not edit files inside `registry/` or `scripts/` unless you intend to customize the framework itself.

## inbox/

The inbox is the **asynchronous communication bus** between agents. Agents never call each other directly вЂ” all handoffs happen by writing a message file and running `post_work`.

### Flow

```
Agent writes draft/message.md
  в†’ runs post_work/main.py
    в†’ unread/ contents archived to read/{timestamp}_{from}_{to}/
    в†’ draft/ contents promoted to unread/
      в†’ Orchestrator runs scan_inbox/main.py
        в†’ reads to: field в†’ invokes next agent
```

### Folders

| Folder | Purpose |
|---|---|
| `draft/` | The agent writes `message.md` (and any attachments) here before calling `post_work`. |
| `unread/` | The active message being processed. Only one message lives here at a time. |
| `read/` | Immutable archive. Every processed message is preserved in a subfolder named `{YYYYMMDD_HHMMSS}_{from}_{to}/`. |
| `internal_templates/` | Copies of `message_briefing.md` and `message_report.md` for agents to reference when composing messages. |

### Message Format

Every `message.md` must begin with a `<message_metadata>` block:

```
<message_metadata>
from: sender-slug
to: recipient-slug
subject: Brief description
</message_metadata>

Message body here...
```

`post_work` will reject the message and print an error if `from`, `to`, or `subject` are missing or empty.

## memory/

Persistent files shared across all agents throughout the project lifecycle.

| File | Purpose |
|---|---|
| `tech_stack.md` | Canonical record of the project's technology choices. Agents read this before making implementation decisions. |
| `decisions.md` | Append-only log of significant architectural and design decisions made during the project. |

These files are initialized from templates in `registry/internal_templates/` during provisioning and are **never wiped** by subsequent re-provisioning runs.

## scripts/

All scripts are run from the **project root** (not from the script's own directory).

### `scripts/provisioner/main.py` вЂ” Provisioner

Performs a **full wipe and rebuild** of the runtime environment for a given team.

**Input:** `scripts/provisioner/input.json`
```json
{
  "chosen_team": "example"
}
```

**What it does:**
1. Deletes `.roo/`, `.roomodes`, and `memory/`.
2. Recreates `inbox/` with `draft/`, `unread/`, and `read/` subfolders.
3. Reads `registry/teams/{chosen_team}/agents.json` to get the agent roster.
4. For each agent, traverses the rules inheritance hierarchy and copies the flattened rule files into `.roo/rules-{slug}/`.
5. Injects the team's `workflow.md` into every operational agent's rules directory (skipped for `apd-orchestrator`).
6. Builds `.roomodes` from each agent's `mode.json`.
7. Initializes `memory/tech_stack.md` and `memory/decisions.md` from internal templates.
8. Copies message templates into `inbox/internal_templates/`.
9. Auto-resets `input.json` to `{"chosen_team": ""}` after completion.

**When to run:** Automatically called by `adp-headhunter` after team selection. Can also be run manually to re-provision the environment.

> **Warning:** The provisioner wipes `.roo/`, `.roomodes`, and `memory/`. Any message currently in `inbox/unread/` is **not** wiped, but `memory/` is reset.

---

### `scripts/utils/post_work/main.py` вЂ” Post-Work

Validates the outgoing draft message, archives the current `unread/` contents, and promotes `draft/` to `unread/`. Every operational agent **must** run this before outputting `Done`.

**Steps:**
1. Reads and parses `inbox/draft/message.md`.
2. Validates that `from`, `to`, and `subject` are present and non-empty.
   - On failure: prints a descriptive error and exits without moving anything. Fix the draft and re-run.
3. Archives `inbox/unread/` contents into `inbox/read/{YYYYMMDD_HHMMSS}_{from}_{to}/`.
4. Moves all files from `inbox/draft/` into `inbox/unread/`.

```bash
python agent_framework/scripts/utils/post_work/main.py
```

---

### `scripts/utils/scan_inbox/main.py` вЂ” Scan Inbox

Reads the `to:` field from `inbox/unread/message.md` and prints the result to stdout. Called by `apd-orchestrator` at the start of every loop iteration.

**Output (one of three values):**

| Output | Meaning |
|---|---|
| `user` | Message exists and `to: user` вЂ” Orchestrator pauses for human intervention |
| `{agent-slug}` | Message exists and `to: {slug}` вЂ” Orchestrator invokes that agent |
| `empty` | No `unread/message.md`, or the file has no `to:` field |

```bash
python agent_framework/scripts/utils/scan_inbox/main.py
```

## registry/

The registry contains all agent blueprints and team definitions. It is the **source of truth** for the provisioner.

### `registry/agents/`

Hierarchical directory of rule files (`.md`) and mode definitions (`mode.json`). The provisioner traverses this hierarchy to assemble each agent's effective ruleset:

```
agents/common/                              в†ђ global rules (all agents)
в”њв”Ђв”Ђ agents/fixed/common/                    в†ђ fixed-agent shared rules
в”‚   в””в”Ђв”Ђ agents/fixed/adp-orchestrator/      в†ђ orchestrator-specific instructions + mode.json
в””в”Ђв”Ђ agents/operational/common/              в†ђ operational agent rules
    в”њв”Ђв”Ђ agents/operational/{domain}/common/ в†ђ domain-shared rules
    в”‚   в””в”Ђв”Ђ agents/operational/{domain}/{agent}/ в†ђ agent-specific instructions + mode.json
    в””в”Ђв”Ђ agents/operational/initialization/common/
        в””в”Ђв”Ђ agents/operational/initialization/adp-headhunter/
```

### `registry/teams/`

Each subdirectory defines a team:

```
registry/teams/
в”њв”Ђв”Ђ init/
в”‚   в””в”Ђв”Ђ agents.json       в†ђ bootstrap team (orchestrator + headhunter only)
в””в”Ђв”Ђ example/
    в”њв”Ђв”Ђ agents.json        в†ђ team roster
    в””в”Ђв”Ђ workflow.md        в†ђ routing table injected into every operational agent
```

**`agents.json` format:**
```json
{
  "team_name": "Example Team",
  "agents": [
    { "slug": "apd-orchestrator", "path": "fixed/apd-orchestrator" },
    { "slug": "apd-example-coder", "path": "operational/development/apd-example-coder" }
  ]
}
```

**`workflow.md` format:**
```markdown
<routing_table>

| Trigger Condition | From | To |
|---|---|---|
| Feature Implemented | apd-example-coder | apd-example-tester |
| Tests Passed        | apd-example-tester | user |

</routing_table>
```
